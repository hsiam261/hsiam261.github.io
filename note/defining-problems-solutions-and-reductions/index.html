Defining Problems, Solutions and Reductions

Apr 05, 2025
note

How Do You Define Problems?

Consider the scenario that you are in an interview and the interviewer asked you a question $q$ from a selected pool of questions $Q$. And you reply with a response $r$. If the response $r$ is deemed appropriate for the question $q$ by the interviewer, then you got the answer right. There might be one or more responses that the interviewer deems appropriate. Our goal is to construct a way to pass the interview regardless what question the interviewer selects from $Q$.

We can abstractly model this scenario as a relation $P$ between two sets $Q$ and $R$. Where $Q$ is the set of questions and $R$ is the set of well formed responses. Members of $q$ are known as instances of the problem $P$. Given any instance $q \in Q$, we want to find a response $r \in R$ such that $(q, r) \in P$. We may make the simplifying assumption that (we will justify later) for any $q \in Q$, there exists atleast one $r \in R$ such that $(q,r) \in P$.

Not just the interview problem, any well defined problem can be modeled a relation where every element of the source set has atleast one image. And our objective is given an instance of the problem (an element from the source set of the relation) find an image to it.

Problems and Encodings

When source and target sets to a relation are countable (which is always the case in computer science), the inputs and outputs can be encoded with binary strings. Thus we may further restrict our definition of problems to relations from $\{0,1\}^*$ to $\{0,1\}^*$ where every element in $\{0,1\}^*$ has an image.

Note that difficulty of our problem does vary based on our choice of encoding the problem. For example, consider the RSA experiment. In the RSA experiment, you are given a semi-prime number (is the product of exactly two primes) $N$, a number $e < \phi(N)$ co-prime to $\phi(N)$ where $\phi$ is the Euler totient function and an integer $y < N$ comprime to $N$ and you have to compute the $e$-th root of $y$. When the numbers are encoded as binary strings, the RSA assumption says that for some distribution of inputs this problem is hard even in the average case. However, instead of representing numbers with binary encoding, if we represented them with their prime factorization (encoded in binary), then problem becomes easy.

Thus the choice of encoding may make the problem very easy or difficult or outright impossible.

Another issue that may arise with encodings that are invalid. Not all binary strings actually encode a valid input to our original problem. For example, we may choose to encode a graph with $n$ vertices with a encoding of its size concatenated by an encoding of its adjacency matrix. We will encode the number $0$ as $00$, the number $1$ as $11$ and the number $01$ as the separator between the number of vertices of the graph and its adjaceny matrix. With such an encoding the string $11000100111100$ encodes $K_2$. Notice that not all binary strings are valid encodings to our problem. As an example, the number $0$ does not encode any valid graph with such an encoding.

If such a case occurs where we not all binary strings are valid encodings, we will have to decide what should be considered as an acceptable answer when provided a binary string that doesn’t encode any valid input. One approach in such a scenario may be that we want to be notified when that happens. In that regard, we will have fix some binary string to encode the result that the input binary string is invalid. This is why we can get away with adding the restriction that every instance of our problem has atleast one image/acceptable answer.

This is not the only way we get to resolve the issue of invalid encoding. Another approach may be given an invalid encoding we accept any response as a valid response since the encoding is invalid. In that regard, we may update our encoded problems relation to reflect that. (Note that for a Turing Machine to be solution to our problem, we will still need it to halt for invalid strings).

What If We Want to Relax The Restriction On Halting For Invalid Inputs?

We may consider doing that. But in the next section we show that a solution in this relax variant of the problem, leads to an encoding that halts on all inputs.

Why We Ensure That Every Instance Will Have An Image For Solvable Problems?

Consider a relation $P$ from $Q$ to $R$ where $Q$ and $R$ are countable sets. Lets assume that there exists an encoding $E$ of $Q$ and $R$ such that a Turing machine $T$ exists for which the following holds:

  • If $x$ is the encoding of an element of $Q$ that has an image in $P$, then $T(x)$ is an encoding of an image of $x$.
  • If $x$ is not such a binary string, then $T(x)$ may or maynot halt.

Then, we may construct a new encoding for the problem $P$. In this new encoding, we interpret the values of $R$ as the encoding $E$ interprets it. But we interpet values the encoding of instances as follows:

  • We first check if the encoding of the input encodes pair of integers in some set encoding for encoding pairs.
  • Then we check the values of the pair. Let the values be $x$ and $y$. We check if $y$ is a certificate that ensures that $T$ terminates on $x$. Note that such a certificate exists, the computation history of $T$ suffices as such a certificate.
  • Then we interpret $x$ as $E$ interprets instances.

In this encoding, we can solve the problem $P$ when we allow arbitrary answers to encodings that are invalid. Furthermore, in the encoded problem every element of $\{0,1\}^*$ has an image.

Some Special Types of Problems

We look at some special types of problems and how they can be modeled using our definition of a problem:

Decision Problem: A decision problem can be modeled as a graph of a function where the range of the function is $\{0,1\}$.

Function Computation Problem: In such problems we will have to compute a function $f:\{0,1\}^*\to\{0,1\}^*$. This be modeled as a graph of $f$.

Optimization Problem: Given $x \in \{0,1\}^*$, we have to compute $\min_{y \in \{0,1\}^*}f(x,y)$ for some objective function $f(x, \cdot)$. This is just a special case of the function computation problem.

Optimizing Search Problem: Given $x \in \{0,1\}^*$, we have to compute $\argmin_{y \in \{0,1\}^*}f(x,y)$ for some objective function $f(x, \cdot)$. This is just a special case of the function computation problem.

Search Problem: Given $x \in \{0,1\}^*$, find $y$, such that $(x,y)$ is in some relation $P$. This follows the general pattern of how we defined problems.

Approximating Search Problem: Given $x \in \{0,1\}^*$, find $y$, such that $(x,y)$ is in some relation $P$ and the objective value $f(x,y)$ does not deviate too much from $\min_{y \in \{0,1\}^*}f(x,y)$ where $f$ is some objective function. We may model this as a relation $P'$ where $(x,y) \in P' \iff (x,y) \in P \land \text{d}(f(x,y), \min_{y \in \{0,1\}^*}f(x,y)) \text{ is not too high }$ where $d$ is some deviation function (usually division).

Solutions

We say a Turing machine solves a problem modeled by a relation $R$ iff. the output of the turing machine (value in the output tape after machine halts) for an input $x$ is an image of $x$. Since turing machines are deterministic, the mapping from the input $x$ to the output that a turing machine will provide for $x$ is a function. We may call such function $f$ a solution function for the relation $R$.

More formally, let $R$ be a relation from $\{0,1\}^*$ to $\{0,1\}^*$ and $f:\{0,1\}^*\to\{0,1\}^*$ be a function such that $f(x)$ is an image of $x$ in $R$. We call $f$ a solution function for $R$. We say that a turing machine solves $R$ if the turing machine computes $f$ for some solution function $f$.

Reductions

A reduction aims to find a solution to problem $A$ given a blackbox solution of problem $B$.

Formally, an oracle machine is a Turing machine with an access to one or more oracles. An oracle to a function $f:\{0,1\}^*\to\{0,1\}^*$ is a pair of tapes (with their own tape heads) called the input tape and the output tape and a two states: oracle-ask and oracle-return. If a turing machine has an access to an oracle for $f$, it may compute $f(x)$ for some string $x$ by first writing $x$ to the oracle input tape and then transitioning to the oracle-ask state. When an oracle machine is in oracle-ask state for an oracle to some function $f$, the oracle machine writes the answer $f(x)$ to the oracle output tape and moves the oracle output tape head to the start of the output tape, empties input tape and immediately transitions to the oracle return state.

Using oracle machines we may consider solving problems when given access to a blackbox to one or more functions. If there exists an oracle machine that can solve problem $A$, when given access to any solution function $f$ to the problem $B$, we say that problem $A$ reduces to $B$ and the oracle machine is called a reduction. We denote that $A$ reduces to $B$ with $A \leq B$ or $A \leq_{T} B$. Such reductions are also known as Turing Reductions, since if $B$ is computable, we can use the oracle machine that solves $A$ with an access to an oracle to some solution function of $B$ to construct a Turing Machine that solves $A$.

Note that for an oracle machine to be a reduction from problem $A$ to problem $B$ the machine must be able solve problem $A$ regardless of the solution function of $B$ that the oracle can compute.

Many To One Reduction

If we have a Turing reduction from problem $A$ to problem $B$, it means that computability of $B$ implies the computability of $A$. But sometimes, we want to know if two problems are more intimately related. We say that a problem $A$ is homomorphic to problem $B$ there exists a pair of function $h_{in}:\{0,1\}^*\to\{0,1\}^*$ and $h_{out}:\{0,1\}^*\to\{0,1\}^*$ such that $(x,y) \in A \iff (h_{in}(x),h_{out}(y)) \in B$.

There exists a homomorphism from $A$ to $B$, then the problem $A$ is just a subproblem of problem $B$ with a different (not necessarily injective) encoding. If the functions $h_{in}$ and $h_{out}$ are computable, then there exists a Turing reduction from $A$ to $B$ that invokes $B$ exactly once. We may call such a reduction a homomorphic reduction.

A special case of homomorphism appers when we force the function $h_{out}$ to be the identity function. This may look like an artificial restriction, but this makes more sense when we are solving decision problems and function problems.

We have already defined decision problems as a relation where the range is $\{0,1\}$. Another way to view decision problems are as a set $L \subseteq \{0,1\}^*$ and given $x \in \{0,1\}^*$ our objective is to determine whether or not $x \in L$. It’s not hard to see that both definitions are equivalent. But under this new definition our restricted homomorphism of our former definition of decision problems results in homomorphism between the two problems in latter definition.

For function problems, if problem $A$ seeks to compute a function $f$ and problem $B$ seeks to compute a function $g$ and if there is such a restricted homomorphism $h$ from problem $A$ to problem $B$, we can see that $f(x) = g(h(x))$ which is an interesting relation.

If there is a such a restricted homomorphism $h$ from a problem $A$ to problem $B$ and $h$ is computable then then there exists an oracle machine that solves $A$ using an arbitrary oracle of $B$ with exactly one query to the oracle and using the result returned by the oracle verbatim and viceversa. We call such an oracle machine a Many to One Reduction from $A$ to $B$.

Many to One reduction is a special case of Turing reduction and does not make any predictions about computability. But rather look for intimate relations between two problems.

One way many to one reductinos are helpful is in the case of classifying problems in terms of their position in arithmetic hierarchy. If a decision problem $B$ is in $\Sigma_i$ for some integer $i$ and if there is a Turing reduction from a decision problem $A$ to $B$, yet we can not make the claim that $A$ is in $\Sigma_i$. For example, there is a turing reduction from the complement of the halting problem to the halting problem. But since, the Halting problem is not recursive, its complement is not recursively enumerable but the halting problem is.

But if we were to only allow many to one reductions, this problem no longer exists. If there is a many to one reduction from $A$ to $B$ and $B$ is in $\Sigma_i$ or $\Pi_i$ for some $i$, then so is $A$. There is no many to one reductions from the complement of the halting problem to the halting problem.

One reason why we may want to add further restrictions on our reductions is to look for such intimate relations. Another reason to add restrictions further is to distinguish between two problems that are different.

Are Many To One Reductions Strictly Better Than Turing Reductions in Computability Theory?

One may assume that due to restricting reductions between problems in different positions in arithmetic hierarchy, Many To One Reductions are strictly better than Turing Reductions. But that is not the case. There may be Turing reductions that are useful in Computability Theory because, there is no guarantee that two computable languages will have a many to one reduction between them. For example, no non-empty computable language has a many to one reduction to the empty set. But, for Turing reductions, we can guarantee that every computable language reduces to another.

So, the problem with Turing reductions are that they are too permissive and allow reductions from non-recursively enumerable languages to recursively enumerable languages and the problem of many to one reductions are that they may be too restrictive and disallow reductions between two computable languages.

Reductions In Complexity Theory

In complexity theory, we are only concerned with problems that are computable and we add further restrictions on time, memory and accuracy of our solutions. Following are some examples of questions that complexity theory studies:

  • Is a computable problem $A$ polynomial time solvable?
  • Is a computable problem $A$ solvable when only given access to polynomial sized memory?
  • Is a computable problem $A$ solvable in FPT time?
  • Can a computable optimization problem $A$ be approximated up to a factor of $r(|x|)$ in polynomial time?

So far in complexity theory, we have been unable to come to exact answer to any such questioins without making some assumptions about difficulty of a problem. In complexity theory we use reductions to create a web implications about the diffculty of problems and define useful classes of problems and study if one is equal to another or a subset of another.

Different Types of Reductions in Complexity Theory

If we were only concerned with whether or not $B$ is computable, then, Turing reductions may be useful. However, in practice, we are more often concerned with computability under restrictions on runtime, memory, accuracy or some combination of them. Under such considerations, we define different types of reductions, some of which we discuss in this section.

Polynomial Time Solvability: In practice we do not just want to solve a problem $A$, but solve it effeciently. The notion of efficiency that is widely accepted in computer science is the notion of polynomial time solvability. We want to find a Turing machine that solves $A$ in atmost $p(|x|)$ steps for any input $x$ where $p$ is some polynomial.

When considering such restrictions, we use the notion of Polynomial Time Turing Reductions. A Turing Reduction is a polynomial time Turing reduction if it runs atmost for polynomial time steps (and therefore can query the oracles atmost polynomial number of times). If there is a polynomial time Turing reduction from $A$ to $B$ and if $B$ is polynomial time solvable, then so is $A$.

Polynomial time Turing reduction is also known as Cook reduction and is denoted as $A \leq_{T}^P B$.

Karp Reduction: Karp reductions are many to one reductions under the restriction that it runs for polynomial times. Karp Reductions are also known as polynomial time many to one reductions and is denoted by $A \leq_{m}^P B$ or $A \leq_{p} B$.

Karp Reductions have the same benefit over Cook Reductions that Many to One reductions have on Turing reductions. Karp reductions are more restrictive than cook reductions and thus allow for more finding more intimate relationship between problems and distinguish between problems and complexity classes that may not be similar. For example, it is not known whether given an NDT that runs in polynomial time, there exists another that recongizes its complement in polynomial time. This is known as the $\mathcal{NP}$ vs $\mathcal{CoNP}$ problem. While Karp Reudcitons may be able to distinguish between $\mathcal{NP}$ and $\mathcal{CoNP}$, Cook reductions can not.

Similar to the case with comutability, with Cook Reduction, every problem in $P$ reduces to every other problem in $P$. But it is not the case in Karp reductions. The no non-empty language in $P$ can reduce to the empty language which is also in $P$.

Karp Reductions are the standard in the literature when studying NP and NP completeness as opposed to Cook reductions.

FPT Reduction: We may want to relax our restriction that a decidable problem needs to be solvable in polynomial time. Instead we may parameterize our problem with some integer $k$ and hope that our problem is polynomial time solvable when $k$ is fixed where the polynomial does not depend on $k$. This is known as fixed parameter tractiblilty.

If there is a Karp reduction from a parameterized problem $A$ to a parameterized problem $B$ such that the following holds:

  • running time of the reduction is FPT
  • the mapped parameter to problem $B$ only depends on the parameter of problem $A$
  • the size of mapped non-parameter input to problem $B$ is bounded by some polynomial of the non-parameter input to problem $A$ then, we call such a reduction an FPT reduction. If there is an FPT reduction from $A$ to $B$ and $B$ is fixed parameter tractible, then so is $A$.

Approximation Preserving Reduction: Often when faced with a computable problem that we have not been able to solve efficiently, we seek to find an approximate solution. Then we may want to solve an approximating variant of the problem in polynomial time. Then we need to add restrictions on reductions so that a polynomial time approximation algorithm for problem $B$ leads to a polynomial time approximation algorithm for problem $A$. Classical versions of approximation preserving reductions are polynomial time homomorphic reduction that preserves some guarantees on the approximation ratio.

For example, we call a homomorphic reduction from an approximation problem $A$ to an approximation problem $B$ if for any oracle of $B$, the approximation ratio for the reduction is bounded by the approximation ratio of the oracle.
